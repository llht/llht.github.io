数据挖掘：关联规则挖掘方法比较与分析

发表于 2020-07-11 更新于 2020-09-03 分类于 技术 Valine：
本文字数： 8.7k 阅读时长 ≈ 8 分钟

本文所使用的数据集以及对应算法实现的代码：数据集及源码

问题简介

本文基于两种独立的数据集，分析和比较了 Apriori 算法，FP-Growth 算法以及一个简单的 Brute-Force 算法在关联规则挖掘应用中的使用效果，同时应用关联规则挖掘算法所得出的结果，分析了一些比较有趣的关联规则，并分别基于这两种数据集提出了一些自己的见解与看法。

数据集介绍

数据集一：杂货店商品成交记录

数据集一记录了一家杂货店在一段时间中多达 9835 次的成交记录，保存在.csv格式文件中，每一条记录包括成交序数以及成交商品类型，典型的成交记录示例如下：

第114条成交记录显示，该次交易商品类型为berries以及whipped/sour cream，第115条成交记录显示，该次交易的商品类型为citrus fruit、berries、UHT-milk以及processed cheese。

数据集二：Unix命令使用记录

第二个数据集包括九个不同的数据子集，它们来自普渡大学某门教学课程中，八个不同同学使用Unix系统命令的记录，其中前两个子集的数据来自同一个使用者，其他七个子集分别属于不同的使用者。该数据集已经被预处理，删掉了所有专用名词（网站地址、文件名等）（并用<数字>来替代，其中“数字”为连续出现的次数），只保留了命令名、命令选项、SHELL元字符。每一个 SHELL Session 都以SOF开头，以EOF结尾。一个简单的示例如下：

挖掘方法

Brute-Force 算法

这应该是直觉上最简单粗暴的一种算法，首先遍历一遍数据集，得到所有 Transaction 中 items 的类型，然后依次举出所有 items 的组合情况（每个组合包含一个item、包含两个item、依次类推），基于每一种组合，遍历一遍数据集计数这种组合的数量。在得到所有组合的数量后，将他们排序，高于最低支持度的所有组合即是我们想要的频繁项集。

Apriori 算法

简单描述，Apriori 就是基于一个先验条件对每一次迭代找出的频繁项集进行剪枝，从而降低后续寻找频繁项集的消耗，最基本的一个先验条件是：频繁项集的非空子集，必定也是频繁项集。我们利用这个基本的先验条件，利用已经得到的 n 频繁项集进行集合并操作构造出 n+1 项集，而不需要对全部 item 进行重新组合，从而大大降低了后续寻找频繁项集的过程中遍历数据库的次数。当然 Apriori 算法在提出后产生了许多变体并衍生出更多的先验条件，但我这里只以这个最基本的先验条件作为 Apriori 算法的核心。

FP-Growth 算法

FP-Growth（Frequent Pattern Growth）算法利用一个从前缀树扩展而来的 FP-Tree 的数据结构，大大降低了扫描数据库的次数，全程只需要扫描两遍。说明算法流程可能需要大量的图文结合，而且课程Slides也已经展现的比较清晰。所以我这里用比较简单的语言来描述这个算法：首先，扫描一遍数据库得到所有的1频繁项集，将数据库每一个transaction按照频繁项集的序号排好序后，再扫描一遍数据库将压缩后的数据库存储到 FP-Tree 的数据结构中，同时得到一个头表作为算法的条件数据库，每一个条件数据库都与一个频繁项集一一对应，接着，对每一个这种数据库递归进行 FP-Growth 算法，最后得到一个单枝的 FP-Tree，这一支就是我们最终得到的频繁项集。

src/Apriori/apriori.py

该部分代码来自于 Github 开源仓库：asaini/Apriori，我只对其中的文件信息读取（dataFromFile）函数进行了一定修改以满足文件格式的要求

以下部分是我对原开源仓库关键代码的解读

首先，输入文件中的每一个transaction信息都经过dataFromFile函数转化为一个frozenset，从而滤去数据集中的无关信息（Grocery中的交易序数，Unix中的SOF、EOF等）。

接着，在主函数runApriori中，将上一步得到的过滤后的数据集近一步处理，得到所有单个item的集合itemSet以及所有transaction组成的二维列表transactionList。

然后，调用returnItemWithMinSupport函数，得到只由1个item组成的频繁项集oneCSet，接着就是不断的迭代了，在runApriori函数的85-99行中，currentLset是由上一阶频繁项集进行并集操作得到的下一阶可能的频繁项集的集合（即item的数量加一），然后调用returnItemsWithMinSupport函数遍历所有transaction，剔除掉currentLset中不频繁的项集。将所有剔除后的集合保存到largeSet中，即得到了我们想要的所有频繁项集。

得到关联规则的算法较为简单，在runApriori函数的111-127行中，遍历每一个item数量大于2的频繁项集（value），对每一个这种频繁项集取全部非空子集，计算得到从子集（element）到互补子集（remain）的置信度，如果满足最小置信度就保存在toRetRules中，这部分的代码的详细注释我已标注在代码中。

使用方法：

终端输入python3 apriori.py -f <path-togroceryorunix.csv> -t grocery -s 0.07 -c 0.1，其中-f指明数据库位置，-t指明数据库类型（grocery或unix），-s为支持度，-c为置信度，即可得到所有可以找到的频繁项集、关联规则以及挖掘所需时间

src/FP-Growth/FP-Growth.py

该部分代码使用了 Github 的开源模块：evandempsey/fp-growth（import pyfpgrowth）。在此份挖掘代码中，我仅对数据集文件进行了一定的预处理（增加了dataFromFile函数）以满足数据集输入格式的要求

以下部分解读模块pyfpgrowth代码实现中的关键部分，该部分代码可以在Github上看到

按照之前的算法介绍部分所讲，首先扫描一遍得到所有的1频繁项集，在FPTree类的构造函数__init__中，首先调用find_frequent_items函数，计数所有 item 出现的个数，之后再遍历一遍得到的 item 与对应个数的字典，删掉不满足额定出现次数（support×transaction数量）的item。接着就是利用得到的1频繁项集构建树以及头表了。在build_header_table函数中，构建一个空的头表，包含所有的单频繁项集，在build_fptree函数中，将压缩后的数据库保存在 FP-Tree 的数据结构中。最后，进入子树以及子树的递归构建过程，直到得到一个单枝的树，将树与条件数据库的头表结合即可得到想要的频繁项集。这部分代码在mine_patterns的递归结构以及mine_sub_trees中已经有很清晰的体现，我不再赘述。

关于关联规则挖掘部分，使用方法与 Apriori 算法实现中的方法相同，相关代码在256-267行.

终端输入python3 FP-Growth.py -f <path-togroceryorunix.csv> -t grocery -s 0.07 -c 0.1，其中-f指明数据库位置，-t指明数据库类型（grocery或unix），-s为支持度，-c为置信度，即可得到所有可以找到的频繁项集、关联规则以及挖掘所需时间

方法比较

我将从三种不同的角度比较这些方法在生成频繁项集这一数据挖掘需求上的应用效果。分别是生成频繁项集的数量，挖掘过程中的内存使用以及算法的运算效率。下面的比较将分别进行。

比较之前：The baseline method

在对这些算法进行比较之前，我首先隆重介绍 Brute-Force 算法的实际表现，并以此作为基本的挖掘尝试，我们也能由此看到缺少基本算法支持的数据挖掘工作是很难进行的。

很显然，这个强搜算法根本无法在我想要的时间内计算得到所有可能的频繁项集，但还是值得一试（并不。我以杂货店交易记录（grocery.csv）作为测试数据集，记录了当support=0.01（100/9835）时，生成每一阶频繁项集所需要的时间：

  • 生成所有1频繁项集：0.73秒
  • 生成所有2频繁项集：62.86秒
  • 生成所有n频繁项集：xxx秒？

根据排列组合原则，所有169个 item 可以构成169个项数为1的排列，意味着需要扫描169次数据库，所有169个 item 可以构成 C（169，2）= 14196 个排列，也就是需要扫描 14196 次数据库，当 r = 3，即需要得到三频繁项集时，需要扫描 C（169，3）= 790,244！次数据库，按照扫描15000次需要60秒来计算，仅得到3频繁项集就需要50分钟时间。如果想估计找到所有频繁项集所需时间，可近似得到：
$$
C(n,0) +C(n,1)+ C(n,2)+C(n,n)=2^n\
T = 2^{169}/15000*60(s)\approx9.5×10^{40}(year)
$$
需要这么多年才能计算的到所有的频繁项集，即使我们选择所有交易中最多的 item 数量（:31）作为r的上限，得到的时间结果依然是一个天文数字。从这个看起来相当愚蠢（时间复杂度直奔指数级别）的方法出发，我们可以看到，一个简单的先验条件（Apriori）或是一个较为复杂的数据结构（FP-Tree）能够为我们的挖掘工作效率带来质的提升。

挖掘过程中的内存使用

Python 语言采用的内存垃圾回收机制使得我们很难监测每一步的内存使用情况，但是根据算法本身的属性、使用的数据结构，我们可以简单对内存使用情况或 space complexity 进行估计比较：

  • Apriori 算法实现采用二维列表（list）结构保存了所有的 transaction，在之后的运行过程中，每次迭代都会将新生成的频繁项集放入一个较大的 list 中，并且频繁进行整个数据库的扫描。我们可以近似地认为 Apriori 算法需要频繁地使用较大的内存空间（整个数据库信息）
  • FP-Growth 算法只扫描两次数据库，并把一个压缩后的数据库保存在 FP-Tree 中，我们不需要保存所有的数据库中Transaction的信息，而只需要保存每个 item 对应的出现次数即可，可以说极大的减少了运行过程中所需要的内存空间，<ITEM><出现次数>的配对存储使得存储空间大大压缩，远低于 Apriori 算法所需的运行时空间。

为了保证时间结果的可靠性，我将对于数据的预处理部分也包含在时间统计中，以反映实际生产环境中两种算法的运行效率，使用 Python 语言的 time 模块可以比较轻松的做到这一点。

通过变换支持度，我得到了两种算法分别应用于两个数据库的时间变化情况，并将结果绘制为曲线图。（UNIX部分依旧以USER1为例）

根据曲线图给出的信息，我们可以得到以下结论：

  • 对于一种特定挖掘方法来说，设定的支持度值越低，所需要的挖掘时间越长，且 Apriori 算法所需时间呈近似指数形式增加或减少
  • FP-Growth在几乎所有支持度区间的挖掘效率都要高于 Apriori 算法，这也与其选用的极其巧妙的 FP-Tree 数据结构有关

结论

本文首先对三种不同的关联规则挖掘算法给出了简要介绍，并对其代表性的代码实现进行解读，之后从三个不同角度对他们的应用效果进行了比较，可以得出以下具有代表性的结论：

  1. 在一定的数据量下，使用 Apriori 算法以及 FP-Growth 算法，都可以在较快时间内得到完整数量的频繁项集，从而对关联规则进行进一步分析。巧妙的设计使他们相对于 baseline 算法有相当出色的性能表现。
  2. Apriori 算法相比于 FP-Growth 算法，使用了较多的运行时内存空间，FP-Growth 算法将整个大数据库表示为一个精简的树形数据结构是其内存利用率低的主要原因。
  3. FP-Growth 算法相比于 Apriori 算法有着极其出色的运行效率表现，总能在相对较短时间内得到想要的全部频繁项集，而且在大幅调整支持度时，也只产生了较轻微的性能波动。

在对这些关联规则挖掘算法进行了相对全面的介绍与比较后，本文对算法生成的关联规则们进行了简要分析，分别提出了一些对于杂货店以及UNIX命令使用方面的观察和见解，作为本文的结尾，我提出两条有趣的发现及预测：

  • 杂货店将蔬菜类食品与奶制品这两种产品置于相邻位置，能够为两类食品带来比较高的销量

  • 一些常用的UNIX指令组合伴随着每一个用户，例如，将cd与ls指令进行组合，设定cl指令缩写（alias），或许是可以为每一个UNIX-Like系统用户提升工作效率的选择
